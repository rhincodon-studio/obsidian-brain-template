# /digest — 處理偏好候選池

整理 AI 觀察到的偏好候選，讓使用者確認是否晉升為正式偏好。

## 執行步驟

1. **讀取候選池**
   - 讀取 `System/candidates.md`
   - 解析待確認的候選項目

2. **按優先順序排列**
   - 🔴 高優先（跨多次對話重複）— 先展示
   - 🟡 中優先（活躍期相關）
   - 🟢 低優先（單次觀察）

3. **逐項展示候選**
   ```
   ## 偏好候選 #1 🔴

   **觀察內容：** [偏好描述]
   **來源：** [哪些對話觀察到]
   **出現次數：** X 次

   建議晉升到：
   - [ ] Identity/PROFILE.md（個人偏好）
   - [ ] System/OPERATING_RULES.md（操作規則）

   請選擇：
   1. 確認晉升到 PROFILE
   2. 確認晉升到 OPERATING_RULES
   3. 暫時保留（下次再處理）
   4. 拒絕（不是偏好，刪除）
   ```

4. **執行晉升**
   - 使用者選擇 1 或 2 → 寫入對應檔案
   - 使用者選擇 3 → 保留在候選池
   - 使用者選擇 4 → 移到「已拒絕」區塊

5. **衝突處理**
   - 若新偏好與既有偏好衝突
   - 嘗試情境化：「在 X 情況下用 A，在 Y 情況下用 B」
   - 若無法情境化，請使用者決定

6. **清理候選池**
   - 移除已處理的項目
   - 超過 30 天的 🟢 項目自動降級或刪除

7. **輸出摘要**
   ```
   ## Digest 完成

   - 晉升到 PROFILE：X 項
   - 晉升到 OPERATING_RULES：Y 項
   - 保留：Z 項
   - 拒絕：W 項

   候選池剩餘：N 項
   ```

## 設計原則

- **自我進化**：系統學習使用者偏好
- **使用者是審核者**：AI 觀察，使用者確認

## 晉升格式

### PROFILE.md 格式
```markdown
## [類別]

- [偏好描述] _(added: YYYY-MM-DD)_
```

### OPERATING_RULES.md 格式
```markdown
## [類別]

- [規則描述] _(added: YYYY-MM-DD)_
```

## 注意事項

- 絕不自動晉升，必須使用者確認
- 保留拒絕記錄，避免重複提議
- Identity 變更需要審核流程
